REQUIREMENTS DOCUMENT IMPROVEMENTS GUIDE
How the New Structure Improves Upon the Original
═══════════════════════════════════════════════════════════════════════════

OVERVIEW
════════

This document explains the improvements made to the requirements documentation
and provides guidance on using the new structured format.

ORIGINAL vs. NEW STRUCTURE
═══════════════════════════

Original Format:
────────────────
* Bullet points with sub-bullets
* No requirement IDs
* Mixed levels of detail
* Updates added as addendums
* No formal structure
* Difficult to reference specific requirements

New Format (SRS):
─────────────────
✓ Hierarchical requirement IDs (REQ-XXX)
✓ Formal document structure (IEEE 830-based)
✓ Clear separation of functional/non-functional requirements
✓ Acceptance criteria for each requirement
✓ Priority levels assigned
✓ Traceability matrix
✓ Professional presentation

IMPROVEMENTS IN DETAIL
═══════════════════════

1. REQUIREMENT IDENTIFICATION
──────────────────────────────

OLD:
  * it shall receive serial data sent by test via serial to the PC
  -- data will be received approximately every 0.1 to 1.0 s

NEW:
  REQ-001: Serial Communication [MUST]
    REQ-001.1: Data Reception Rate [MUST]

Benefits:
- Easy to reference ("See REQ-001.1")
- Hierarchical organization
- Clear parent-child relationships
- Traceability to implementation and tests


2. ACCEPTANCE CRITERIA
──────────────────────

OLD:
  -- data will be received approximately every 0.1 to 1.0 s

NEW:
  Acceptance Criteria:
  - System handles data arriving every 0.1 seconds (10 Hz)
  - System handles data arriving every 1.0 seconds (1 Hz)
  - System handles variable data rates within this range
  - No data loss occurs at any supported rate
  
  Test Method:
  - Connect with mock data at 10 Hz → Verify all data received
  - Connect with mock data at 1 Hz → Verify all data received

Benefits:
- Objective pass/fail criteria
- Testable conditions
- Clear verification methods
- No ambiguity


3. PRIORITY LEVELS
──────────────────

OLD:
  (No priorities specified)

NEW:
  REQ-001: Serial Communication [MUST]
  REQ-002.5: User-Requested Save [SHOULD]

Priorities:
  [MUST]   - Critical, system cannot function without it
  [SHOULD] - Important, significant value
  [COULD]  - Desirable, nice to have

Benefits:
- Clear prioritization for development
- Resource allocation guidance
- Trade-off decision support
- Risk management


4. FUNCTIONAL vs. NON-FUNCTIONAL SEPARATION
────────────────────────────────────────────

OLD:
  (All requirements mixed together)

NEW:
  Section 3: Functional Requirements
    - What the system does
    - Features and capabilities
  
  Section 4: Non-Functional Requirements
    - How well the system performs
    - Quality attributes
    - Constraints

Benefits:
- Clearer organization
- Different stakeholder audiences
- Different testing approaches
- Better understanding of system qualities


5. STRUCTURED DATA SPECIFICATIONS
──────────────────────────────────

OLD:
  DTA;31422;182;263;0;793;2238;0;611;0;!
  1. DTA; Test is running...
  2. 31422; no of cycles
  ...

NEW:
  Section 5.1: Serial Data Format
  
  Field 3: Position 1
    Type: Integer (represents mm with implied decimals)
    Description: Position 1 in millimeters
    Conversion: Divide by 100 to get actual mm
    Example: 182 → 1.82 mm

Benefits:
- Formal specification style
- Clear conversion rules
- Examples provided
- Consistent format
- Easy reference


6. REQUIREMENTS TRACEABILITY
─────────────────────────────

OLD:
  (No traceability)

NEW:
  Section 7.1: Requirements Traceability Matrix
  
  Requirement         Module                Test Case       Status
  ──────────────────────────────────────────────────────────────────
  REQ-001            serial_reader.py       TC-001          Implemented
  REQ-001.1          serial_reader.py       TC-001.1        Implemented

Benefits:
- Links requirements to code
- Links requirements to tests
- Implementation status tracking
- Impact analysis for changes
- Completeness verification


7. VERSION CONTROL
──────────────────

OLD:
  "update 05 feb 2026" annotations

NEW:
  Document Information:
    Document Version:    2.0
    Date:                February 5, 2026
    Status:              Approved
  
  Section 7.2: Change Log
    Detailed list of all changes with rationale

Benefits:
- Formal version tracking
- Approval workflow
- Change history
- Rationale documented
- Professional presentation


8. DOCUMENT STRUCTURE
─────────────────────

OLD:
  Flat list of requirements

NEW:
  1. Introduction
     1.1 Purpose
     1.2 Scope
     1.3 Definitions
     ...
  2. Overall Description
  3. Functional Requirements
  4. Non-Functional Requirements
  5. Data Specifications
  6. Interface Requirements
  7. Appendices

Benefits:
- Easy navigation
- Progressive detail
- Standard format (IEEE 830-based)
- Professional presentation
- Comprehensive coverage

REQUIREMENT ID SCHEME
══════════════════════

Format: REQ-XXX.Y

Where:
  REQ     = Requirement prefix
  XXX     = High-level requirement number (001-999)
  .Y      = Sub-requirement number (1-99)

Examples:
  REQ-001       Top-level requirement (Serial Communication)
  REQ-001.1     Sub-requirement (Data Reception Rate)
  REQ-001.2     Sub-requirement (Serial Parameter Configuration)

Non-Functional Requirements:
  REQ-NFR-XXX   Non-functional requirement

Benefits:
- Clear hierarchy
- Unlimited expansion
- Easy sorting
- Memorable IDs

ACCEPTANCE CRITERIA WRITING
════════════════════════════

Good Acceptance Criteria:
─────────────────────────
✓ Objective (measurable)
✓ Testable (can verify)
✓ Clear (no ambiguity)
✓ Complete (covers all aspects)

Example - GOOD:
  Acceptance Criteria:
  - Plot updates complete within 100 ms
  - No GUI freezing during updates
  - Smooth operation with 10,000+ data points

Example - BAD:
  Acceptance Criteria:
  - Plots should be fast
  - GUI should work well

The good example is:
- Measurable (100 ms, 10,000 points)
- Testable (can measure time, count points)
- Clear (specific conditions)

TEST METHOD SPECIFICATION
══════════════════════════

Each requirement includes suggested test method:

Example:
  Test Method:
  - Run test with 10,000+ points
  - Verify all points visible in plots
  - Verify no old data dropped
  - Monitor memory usage remains reasonable

Benefits:
- Guides test creation
- Ensures testability
- Defines verification approach
- Reduces interpretation differences

MAINTAINING THE REQUIREMENTS
═════════════════════════════

When Adding New Requirements:
──────────────────────────────
1. Assign next available REQ-XXX ID
2. Write clear description
3. Define acceptance criteria
4. Specify test method
5. Assign priority level
6. Update traceability matrix
7. Update change log

When Modifying Requirements:
────────────────────────────
1. Update requirement text
2. Update acceptance criteria if needed
3. Update test method if needed
4. Add entry to change log
5. Update document version
6. Update traceability matrix if module changed
7. Review dependent requirements

When Retiring Requirements:
───────────────────────────
1. Mark as "Deprecated" in traceability matrix
2. Add entry to change log
3. Keep in document for history
4. Note replacement requirement if applicable
